# Protocolo Snowball V1

## Regra central

Cada sessão deve aumentar a compreensão atual e reduzir a ajuda necessária nas sessões futuras.

## Regra de produto

Todo dia o aluno deve receber exatamente a próxima ação de maior retorno, sem precisar decidir nada.

Formato humano:

```text
Hoje você vai fazer:

Por quê:
```

O `Por quê` deve ser curto e concreto: próximos episódios mais fáceis, menos dependência de legenda, uso em vida real ou menos fadiga de decisão. Não mostre a engenharia interna.

## Sprint

Analise de 8 a 12 episódios antes de começar:

```text
snowball_engine_analyze_season
```

Selecione itens por valor composto:

```text
importância atual
× recorrência futura
× utilidade comunicativa
× dificuldade auditiva
× lacuna individual
```

## Capital linguístico

Use `SNOWBALL_CAPITAL.json` para acompanhar:

- reconhecimento;
- escuta;
- produção;
- contextos vistos;
- vozes reconhecidas;
- automaticidade;
- valor futuro estimado.

Níveis:

- passivo;
- receptivo;
- ativo;
- automático.

## Dívida de automaticidade

Se reconhecimento e escuta sobem, mas produção e automaticidade ficam para trás:

- reduza itens novos;
- faça consolidação;
- gere transferência;
- use episódios mais fáceis;
- aumente recuperação curta.

## Revisão futura

Antes de um episódio, revise itens que reaparecerão nele:

```text
revisão
→ expressão aparece naturalmente
→ reconhecimento é recompensado
→ memória fortalece
```

## Incidental

Não transforme tudo em flashcard.

Promova um item incidental somente quando houver evidência de:

- reconhecimento repetido;
- inferência correta;
- uso ou produção.

## Flywheel

Métricas principais:

```text
Autonomy Ratio
Compounding Rate
Assistance Decay
Automaticity Debt
```

Se essas métricas não melhorarem, a bola de neve não está acontecendo.

## Teste cego mensal

Objetivo: medir compreensão real sem preparação.

Fluxo:

```text
episódio/trecho novo
→ assistir sem preparação
→ relatar sentido geral
→ relatar detalhes
→ registrar confusões
→ só então explicar
```

## Rotina oral pós-episódio

Depois de cada episódio:

```text
3 respostas faladas
20 a 40 segundos cada
usando chunks do episódio
```

Se o aluno escrever em vez de falar, registre como produção não oral.

## Anti-complexidade

A interface humana padrão tem quatro comandos:

```text
começar
continuar
energia baixa
ver progresso
```

## Modo Vida Real

Série, diálogo e vídeo são laboratórios. O objetivo é fluência funcional geral.

Todo chunk aprendido deve passar por transferência:

```text
chunk no conteúdo
→ frase fora do conteúdo
→ resposta falada
→ situação real simulada
```

Trilhas:

- viagem;
- conversa;
- trabalho/estudo;
- vídeos/séries;
- inglês geral.

## Missão diária

Para TDAH, decisão pesada fica com o agente e execução pequena fica com o aluno.

```text
continuar
→ agente escolhe revisão, chunk e vida real
→ aluno executa uma microação
```

O agente nunca deve perguntar "o que você quer estudar hoje?".

## Placar de fluência funcional

Fechamento diário:

```text
Missão concluída
→ calcular placar
→ mostrar __/100
→ dizer o que melhorou
→ preparar o próximo passo
```

O placar é motivacional e operacional. Não é porcentagem literal de fluência.

## Modo Guerra

Missões de 3 minutos para romper inércia:

```text
falar/gravar 30 segundos
sem explicação
sem teoria
sem painel
```

## Modo Retorno

Após ausência:

```text
sem culpa
sem backlog
2 chunks no máximo
5 minutos
```

## Produção Primeiro

Todo chunk novo:

```text
ver
ouvir
usar
```

Produção acontece na mesma sessão.

## Oral, listening e conversa

Três regras:

```text
1. chunk novo → cinco respostas faladas
2. áudio → primeiro sem legenda
3. conversa → sem múltipla escolha
```

Essas regras têm prioridade sobre métricas internas.

## Capacidades funcionais

O painel humano vem antes do score:

```text
Você já consegue:
✓ pedir ajuda
~ pedir comida

Ainda treinando:
□ contar histórias passadas
```
